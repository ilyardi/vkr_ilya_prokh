# Сервисы планирования и мониторинга ресурсов телекоммуникационной компании
 
Программная реализация ВКР «Разработка сервисов планирования и мониторинга
ресурсов для телекоммуникационной компании». Система реализует встраиваемый
функциональный слой над действующей биллинговой системой LANBilling и
обеспечивает автоматизацию операционной деятельности регионального оператора
связи: ведение реестра задач, планирование временных слотов выездного
персонала, проектный учёт, учёт расходов и автоматизированную категоризацию
абонентов с дебиторской задолженностью.
 
## Архитектура
 
Система построена по клиент-серверной модели с разделением на три уровня:
 
- Клиентский уровень — одностраничное React-приложение, взаимодействующее
  с серверной частью через REST API.
- Серверный уровень — приложение на Ruby on Rails (паттерн MVC), запущенное
  под управлением сервера Puma.
- Уровень хранения данных — две физически разнесённые СУБД:
  - PostgreSQL — основная база, хранит собственные объекты сервисов;
  - MySQL (LANBilling) — биллинговая база, используется только в режиме
    чтения через изолированное подключение.
Все модели, представляющие сущности биллинга, наследуются от базового класса
LbBase, который инкапсулирует подключение ко второй базе данных и не
допускает операций записи.
 
## Структура каталогов
 
Исходный код организован в виде типового Rails-приложения. Структура отражает
классическое разделение на слои: модели предметной области, контроллеры,
представления, фоновые задачи, сервисные классы и клиентские компоненты.

### Слои приложения
 
- `app/models` — классы предметной области: задача (`Task`), проект
  (`Project`), расход (`Expense`), рабочий день (`WorkingDay`), должник
  (`Debtor`), пользователь (`User`) и связанные справочники. Здесь же
  реализованы валидаторы временных пересечений по исполнителю и закреплённому
  транспортному средству, а также механизм дочерних задач (`parent_id`,
  `add_helper`).
- `app/controllers/api/v1` — контроллеры программных интерфейсов,
  обрабатывающие запросы клиентского приложения и формирующие JSON-ответы.
- `app/javascript/bundles` — компоненты пользовательского интерфейса,
  маршрутизация, хранилище состояний и вспомогательные утилиты клиентской
  части.
- `app/services` — сервисные классы, инкапсулирующие прикладную логику,
  не относящуюся к моделям и контроллерам.
- `app/jobs` — фоновые задачи (синхронизация должников, формирование
  служебных задач, обработка вложений).
## Сервисы системы
 
В рамках реализации выделены четыре взаимосвязанных операционных сервиса,
образующих ядро прикладной функциональности:
 
1. Сервис управления задачами (раздел 3.2 ВКР) — операционное ядро
   системы. Реестр задач с панелью фильтров и сохраняемыми шаблонами поиска,
   иерархия родительская/дочерняя задача, автоматическая проверка временных
   пересечений по исполнителю и транспорту, прикрепление файлов с
   группировкой по папкам.
2. Сервис планирования временных слотов / календарь-планировщик
   (раздел 3.3) — визуальное расписание выездного персонала с разграничением
   по ролевой модели. Метод slots_by_week формирует трёхмерную структуру
   «день недели — пользователь — время».
3. Сервис управления проектами — реестр инфраструктурных и коммерческих
   проектов как контейнеров для группы задач, объединённых общей целью и
   сроками. Карточка проекта включает вкладки атрибутов, связанных задач и
   документов.
4. Сервис учёта должников — автоматизированная категоризация абонентов
   с дебиторской задолженностью по четырём типам услуг с переносом статуса
   между расчётными периодами и динамической сверкой баланса.
## Развёртывание
 
Серверная часть оформлена как Docker-контейнер. Production-развёртывание
описано в docker-compose.yml и включает:
 
- контейнер приложения (Rails + Puma);
- контейнер фоновых задач;
- контейнер планировщика;
- контейнер веб-сервера (Nginx).
Входящий трафик обрабатывает Nginx: отдаёт статические файлы клиентской
сборки, проксирует обращения вида /api/* на сервер приложения и обеспечивает
HTTPS с использованием сертификатов Let's Encrypt. Доменное имя
new.teleset.plus указывает на адрес производственного сервера.
 
Клиентская часть собирается Webpacker в набор статических файлов;
монтирование интерфейса в страницу обеспечивается библиотекой react_on_rails.
 
## Технологический стек
 
- Backend: Ruby, Ruby on Rails, Puma
- Frontend: JavaScript, React, Ant Design, Material UI, Webpacker,
  react_on_rails
- СУБД: PostgreSQL (основная), MySQL (LANBilling, read-only)
- Библиотеки: Devise (аутентификация), CanCanCan (авторизация),
  PaperTrail (аудит изменений), ActiveJob (фоновые задачи), Whenever
  (планировщик)
- Инфраструктура: Docker, Nginx, Let's Encrypt
